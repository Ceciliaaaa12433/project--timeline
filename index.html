01 /

为什么做这个需求发现

项目的起点很朴素——我自己每天上班都会焦虑。不是怕迟到，是怕「不知道会不会迟到」。

备忘录手打太慢，普通计时器只给总时长，没法告诉我「现在到底哪个环节超了」。

于是我用 AI 辅助编程，从零独立开发了这款微信小程序——把通勤路径切成可控进度条，让每一步都有数据托底。

心理学洞察

焦虑的本质是对不确定性的失控感。我需要的不是一个闹钟，而是一个能把不确定切成可控节点的工具——在路上安心发呆，到站看一眼提示就行。

02 /

V1：先跑起来，再说对不对MVP 阶段

MVP PRINCIPLE 先砍掉所有「可以有」，只留「必须有」。固定单条路线 + 6个预设节点，不是偷懒，是有意识地降低技术死角，确保核心体验完整落地。
← 我的小程序截图 →
FEAT · 01
⏱ 顺序解锁计时
按节点顺序点击，自动记录时间戳并计算相邻差值（如「步行 4 分 32 秒」）
FEAT · 02
🔄 防假死动态读秒
基于系统时间戳计算，切后台不丢精度——「如果卡住了我会知道」
FEAT · 03
🗺 场景化即时提示
到达特定节点后激活「本面出 / 对面出」导航文本，只在需要时出现
FEAT · 04
↩ 防误触撤销
支持回退上一步并清除对应记录
FEAT · 05
📋 弹窗结算 + 静默存档
点击终点弹出完整数据卡片，同步写入本地历史。强化完成仪式感，像打通游戏关卡。
FEAT · 06
📜 极简历史记录
纯文本列表，倒序展示，不可编辑——「错了就错了」，符合用完即走原则
░░░ DESIGN DECISIONS ░░░
设计决策 背后的理由
固定路线，不做自定义 降低技术复杂度，确保 MVP 快速可用，避免技术死角
历史记录不可改 符合「用完即走」原则，规避编辑态逻辑复杂度
弹窗式结算 强化完成仪式感，像打通游戏关卡——情绪设计
微信原生而非网页 愿意承受配置摩擦，换取最流畅的原生体验
手动截图，不做自动生图 规避跨设备渲染与相册权限 Bug，0 开发风险
░░░ V1 LIMITS ░░░
🚧
只有一条路线——现实通勤有变数（雨天走地下通道、换出口、偶尔绕路），V1 完全无法应对
📊
无多路线对比——每天只能看「今天用了多久」，无法横向比较不同策略，这才是真正的核心场景
🎨
UI 经历两轮试错——从现代扁平渐变，到 90 年代高饱和复古，最终妥协到低饱和怀旧像素风
03 /

V1 → V2：从 MVP 到核心业务需求驱动迭代

不是因为用户抱怨，是因为我自己用着不够用。
真实通勤不是一条线，是一张图。单一路线的设计在概念验证阶段够用，但无法支撑「每天对比、选最优策略」这个真正的核心场景。

← 我的小程序的V1与V2版本的并排对比截图 →
NEW · 01
路线架构重构
  • 支持自定义多条路线，独立路线库管理
  • 「横线搭积木」式节点拼装
  • 完整 CRUD：设为当前 / 重命名 / 删除
NEW · 02
动态适配首页
  • 时间轴根据节点数量动态渲染
  • 卡片新增节点序列文本摘要
  • 像素风方向虚线连接节点
NEW · 03
交互防呆升级
  • 操作台下沉，适配右手热区
  • 保存路线自动跳转首页
  • 未命名路线自动填充编号
NEW · 04
情绪备忘
  • 历史记录页开放自定义情绪输入
  • 数据不只是数字，有温度
  • 「在路上」取代「上班」——去班味
PM 视角 · 技术亮点

全程基于自然语言指令协同 Cursor 进行 AI 辅助编程。核心挑战不在于「写代码」，而在于准确描述产品行为、拆解逻辑边界、复现和定位 Bug——这与 PM 的核心能力高度重合。

BUG 01 白屏 Bug：定位到跨页面 Storage 状态未级联更新，重构数据流转逻辑
BUG 02 视口锁死 / 内容溢出：引入 min-height: 100vh + 全面屏安全区适配
BUG 03 「平行宇宙 Bug」：在两个本地文件夹之间改错版本，深刻认识版本管理的必要性，确认接入 Git 的优先级

05 /

反思与下一步

01
需求不是调研出来的，是活出来的
自用倒逼了最真实的痛点发现。没有用户访谈，但有每天真实使用的数据和体感。
02
设计决策需要有理由
每一个「砍掉」和「保留」都对应了具体的使用场景和约束，而不是凭感觉。
03
迭代要有主线
V2 的所有改动都围绕「多路线策略」这一个核心需求展开，没有发散，没有为了加功能而加功能。
⚠
诚实的局限
没有真实多用户数据，无法验证「通勤焦虑」是否是普遍痛点，也无法用数据衡量版本迭代的实际效果。
▶ 如果有真实用户，下一步想验证的假设

多路线对比功能，是否真的降低了用户的决策焦虑——而不只是增加了操作步骤？
